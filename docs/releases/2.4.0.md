# Super Dev v2.4.0

发布日期：2026-04-22

## 版本定位

`2.4.0` 是一次真正的产品面收口版本。

目标不是继续堆内部治理能力，而是把 `Super Dev` 拉回一个更清晰、更强、更可发布的商业产品形态：

- 终端极简
- 宿主优先
- 首页与快速开始回到 pip 主安装口径
- 安装后 5 分钟路径更像成熟产品，而不是一组散乱命令
- 已有项目可恢复、可迭代、可审计
- 宿主适配统一成协议层
- UI 设计链真正开始反模板化、反 AI 味

这版开始明确收正产品身份：

- 宿主负责执行
- `Super Dev` 负责图纸、节奏、审美、门禁、验收与交付标准
- `Super Dev` 不做模型平台，也不把“本地生成代码”当主卖点

---

## 重点更新

### 1. 公开产品面正式收回宿主优先模型

- 终端公开命令正式固定为：
  - `super-dev`
  - `super-dev update`
  - `super-dev uninstall`
- 宿主公开主入口固定为：
  - `/super-dev <goal>`
  - `/super-dev-work <mode> <goal>`
  - `/super-dev-run <stage|resume|status|next>`
  - `/super-dev-review <target> <action>`
  - `/super-dev-seeai <goal>`
- 普通用户不再需要把 `doctor / detect / integrate / release / quality / spec / task` 当主路径来学。
- `super-dev-core` 从现行代码、模板、技能、测试合同中清退，当前产品名统一只保留 `super-dev`。

### 2. 已有项目主链和恢复能力补齐

- `work_mode` 正式进入状态模型：
  - `new`
  - `evolve`
  - `variant`
  - `patch`
  - `resume`
- 已有项目主链正式收敛为：
  - `baseline -> baseline confirmation -> delta research -> docs -> docs_confirm -> spec -> frontend -> preview_confirm -> backend -> quality -> delivery`
- 新增正式 baseline 工件与治理真源：
  - `output/*-baseline-audit.md`
  - `output/*-baseline-audit.json`
- `resume_gate` 与 `workflow_context` 进入：
  - `detect`
  - `doctor`
  - `validate`
  - `proof-pack`
  - `release-readiness`
  - `quality`
- 关窗口、关电脑、第二天继续，恢复不再被视为异常场景，而是默认产品路径。

### 3. 宿主接入升级为统一宿主矩阵

- 当前 canonical host matrix 为：
  - CLI `12`
  - IDE `9`
  - 桌面助手 `5`
- `Droid CLI`、`Kimi Code`、`Qwen Code`、`Trae SOLO`、`Trae SOLOCN` 正式进入一等宿主矩阵。
- `OpenClaw` 从现行支持面退出。
- 宿主适配不再只是散落文案，而是进入统一协议层：
  - `adaptation_contract`
  - `official_alignment`
  - `host_experience_profile`
  - `host_usage_profile`
  - `host_start_playbook`
  - `host_resume_guidance`
  - `host_repair_playbook`
  - `post_onboard_self_check`
  - `official_workflow_checks`
- `standard_flow_ready` / `competition_flow_ready` / `injection_closure` 现在贯通到：
  - CLI
  - Web API
  - catalog
  - decision card
  - runtime markdown report
  - installer

### 4. 安装器与生命周期结果开始产品化

- 首页 / Quick Start / 安装说明改成明确的 `uv tool install super-dev` 主口径。
- `uv` 成为统一的默认安装路径。
- 安装完成后现在会自动落盘：
  - `output/maintenance/host-onboard-smoke-*.json`
  - `output/maintenance/host-onboard-smoke-*.md`
- smoke guide 会直接告诉用户：
  - 标准流第一句
  - 比赛流第一句
  - 框架焦点（Framework Coaching Focus）
  - 必验场景与交付证据
  - 接入后先验
  - 官方工作流检查
  - 恢复方式与修复优先级
- 这让“安装后 5 分钟应该做什么”第一次变成稳定产品路径：
  - 安装
  - 选宿主
  - 看首句
  - 看框架焦点
  - 跑 host-onboard-smoke
  - 回宿主真正开工
- `super-dev uninstall` 现在支持：
  - `super-dev uninstall --dry-run`
  - `super-dev uninstall --dry-run --json`
- 卸载与预演都会落盘：
  - `output/maintenance/host-cleanup-*.json`
  - `output/maintenance/host-cleanup-*.md`
- 这意味着安装、接入、验收、清理第一次形成了更像成熟商业产品的完整生命周期，而不是一组离散脚本。

### 5. 接入默认改为项目优先

- 默认接入行为正式改为：
  - 先写项目级接入面
  - 用户级 / 全局级接入面必须显式 `--with-user-surfaces`
- 不再默认把总则或规则写进系统级 `AGENTS.md`。
- 当前默认行为更符合真实团队协作：
  - 项目仓库内生效
  - 不污染全局
  - 用户级接入改为显式 opt-in

### 6. UI 设计链升级为“反 AI 味”合同

这一版开始把 UI 生成从“会冻结结构”推进到“会先确定视觉方向”，并把“过平、过空、模板味太重”的页面正式拉进阻断链。

新增正式 UI 合同层：

- `art_direction_candidates`
- `design_direction_manifest`
- `anti_ai_slop_guardrails`
- `critique_rubric`
- `tweak_categories`

它们已经进入：

- `output/*-uiux.md`
- `output/*-ui-contract.json`
- 前端 blueprint / frontend builder
- `ui-review`
- `quality-gate`

并且新增的截图级视觉信号现在会继续流入：

- `release-readiness`
- `proof-pack`
- 宿主 runtime 报告与安装后 smoke guide

这意味着 UI 不再只是“有没有合同”，而是正式开始检查：

- 页面是否过平
- 是否过空
- 是否过于单一
- 是否仍然像模板页 / AI 占位稿

也就是说，UI 现在不仅会在评审里被指出问题，还会继续影响：

- 安装后 smoke guide 的通过标准
- quality gate 的 UI 契约执行结果
- proof-pack / release readiness 的交付闭环

这意味着 UI 系统开始正式检查：

- 是否有可比较的视觉方向候选
- 是否冻结主视觉哲学
- 是否明确禁止模板化 / cliché / “AI 味” 设计
- 是否用五维设计批评标准审查产出

### 7. SEEAI 比赛模式闭环继续补强

- `super-dev-seeai` 的项目级补充面和用户级补充面都进入正式接入闭环。
- 现在系统会明确区分：
  - 标准流 ready
  - SEEAI 项目补充面 ready
  - SEEAI 用户级补充面 ready
- 比赛模式不再只是“支持”，而是有正式 install / validate / report / catalog 信号。
- `doctor / validate / decision card / install.sh` 都会把比赛模式补充面当成正式闭环信号，而不是隐藏补充说明。

## 对用户意味着什么

### 普通用户

- 只需要记住 3 个终端命令：
  - `super-dev`
  - `super-dev update`
  - `super-dev uninstall`
- 默认安装方式就是 `uv tool install super-dev`。
- 接入完成后，所有真实开发都回到宿主里继续。
- 已有项目会先做 baseline，不再假装可以直接编码。
- 恢复成为默认场景，不需要再自己猜下一步。
- 安装后先看 `host-onboard-smoke`，而不是只看终端输出是否“写入成功”。

### 维护者

- 宿主适配不再靠散乱规则文件判断，而是有正式 readiness / alignment / self-check / repair protocol。
- `doctor / detect / validate` 已经能更清楚地区分：
  - 只是推荐宿主
  - 已完成接入
  - 标准流可直接开工
  - SEEAI 比赛流可直接开工

### 设计与前端阶段

- UI/UX 不再只是“写一个安全模板页”。
- 现在系统会要求先冻结视觉方向，再推进页面骨架和前端实现。
- 对“AI 味”“模板感”“品牌弱”“层级弱”的 UI 开始有正式合同和审查语义。
